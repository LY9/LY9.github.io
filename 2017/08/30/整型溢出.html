整型溢出

发表于 2017-08-30 | 更新于 2017-12-15 | 分类于 技术杂谈 |
字数统计 | 阅读时长

手机助手的清理模块有一个功能需要获取存储空间使用比例.

网上找的方法大概是,使用时传入sd卡根目录:

    public static long[] getTotalAndAvailableBytes(String path) {
    if (TextUtils.isEmpty(path)) {
        return null;
    }
    try {
        File pathFile = new File(path);
        StatFs stat = new StatFs(pathFile.getAbsolutePath());
        if (Build.VERSION.SDK_INT >= 18) {
            long total = stat.getBlockCountLong() * stat.getBlockSizeLong();    //磁盘总大小,单位byte
            long available = stat.getAvailableBlocksLong() * stat.getBlockSizeLong();   //磁盘剩余空间,单位byte
            return new long[] {total, available};
        }
        long total = stat.getBlockCount() * stat.getBlockSize();
        long available = stat.getAvailableBlocks() * stat.getBlockSize();
        return new long[] {total, available};
    } catch (Exception e) {
        return null;
    }
}

粗看,已经是做了版本适配的方法,挺靠谱的.

然而在api为17的手机上,出现了存储空间剩余占比大于1的情况,初步猜测就是溢出了.

api为17的情况下的关键代码:

long total = stat.getBlockCount() * stat.getBlockSize();    //磁盘总大小,单位byte
long available = stat.getAvailableBlocks() * stat.getBlockSize();   //磁盘剩余空间,单位byte
return new long[] {total, available};

打印了一下异常时的一些数据:

stat.getBlockCount() = 3217442
stat.getAvailableBlocks() = 2698401
stat.getBlockSize() = 4096
total = 293740544
available = -1832251392

总空间与剩余空间都是long类型,long的最大值9223372036854775807,如果单位是byte,约为8388608TB,讲道理是足够计算手机的存储空间了.

那问题一定出在了看似正确的代码,其实并没有按我们想的去执行.事实也确实如此:

以下面的代码为例:
long total = stat.getBlockCount() * stat.getBlockSize(); 
total为long类型
stat.getBlockCount()为int类型
stat.getBlockSize()为int类型
表达式先进行int相乘(溢出),然后溢出后的int值强制转型为long类型.

解决办法也很简单,在乘法之前先转型:

long total = ((long)stat.getBlockCount()) * stat.getBlockSize();

以上.
